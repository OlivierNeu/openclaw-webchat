# OpenClaw WebChat

Professional WebSocket-native webchat bridge for OpenClaw.

OpenClaw is event-driven and works best through its Gateway WebSocket. This
project provides a secure backend bridge that can be consumed by the bundled
React frontend or by any compatible third-party frontend.

## Status

Early implementation, designed as a public, forkable foundation. The current
backend protocol is intentionally documented and versioned, but still `0.x`.
Breaking changes can happen before `1.0`.

## Why This Exists

Open WebUI pipes are request/response adapters. Once a pipe returns, it stops
listening. OpenClaw can continue producing events after that point: follow-on
runs, delayed tool output, generated media, compaction restarts and messages
after browser reconnects.

This project keeps a persistent WebSocket bridge between the browser and
OpenClaw, then reconciles history through OpenClaw itself.

## Features

- Firebase Google authentication with server-side token verification.
- Multi-instance OpenClaw routing per authenticated user.
- Stable browser-facing WebSocket contract.
- `chat.history` reconciliation after refresh or reconnect.
- `sessions.patch` with `verboseLevel: "full"` before `chat.send`.
- Signed media links for files generated by OpenClaw.
- Local path sanitization before data reaches the browser.
- Optional all-in-one Docker image serving frontend and backend.
- Firebase Hosting compatible frontend deployment.

## Backend Contract

The bridge is the primary product. The bundled frontend is a reference client.

Important endpoints:

- `GET /health`
- `GET /api/capabilities`
- `GET /api/me`
- `GET /api/media/outbound/{filename}?scope={scope}&exp={expiresAt}&fp={fingerprint}&sig={signature}`
- `WS /ws/chats/{chatId}`

See [Bridge Protocol](docs/BRIDGE_PROTOCOL.md) for the WebSocket message
contract.

## Quick Start: Backend

```bash
cd openclaw-webchat/backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
pip install pytest pytest-asyncio

cp ../config.example.json ../config.local.json
export OPENCLAW_WEBCHAT_CONFIG_FILE="$PWD/../config.local.json"
export ALLOW_DEV_AUTH=true
export OPENCLAW_MEDIA_LINK_SECRET="$(openssl rand -hex 32)"
export OPENCLAW_MEDIA_OUTBOUND_DIR="/home/node/.openclaw/media/outbound"

uvicorn app.main:app --reload --port 8080
```

With `ALLOW_DEV_AUTH=true`, the backend accepts `Authorization: Bearer
dev:<email>` and `?idToken=dev:<email>`. Never enable this in production.

## Quick Start: Frontend

```bash
cd openclaw-webchat/frontend
npm install
cp .env.example .env.local
npm run dev
```

For local dev without Firebase, set:

```bash
VITE_DEV_ID_TOKEN=dev:olivier@lacneu.com
VITE_OPENCLAW_BRIDGE_WS_URL=ws://localhost:8080
```

## Deployment Options

### Firebase Frontend + Synology Backend

Use Firebase Hosting for the static frontend and Synology Container Manager for
the backend bridge.

The frontend must be built with:

```bash
VITE_OPENCLAW_BRIDGE_WS_URL=wss://openclaw-webchat-api.example.com npm run build
```

### Synology All-In-One Docker Image

The root `Dockerfile` builds the React/Vite frontend and embeds it into the
FastAPI backend. Leave `VITE_OPENCLAW_BRIDGE_WS_URL` empty so the browser uses
the same host for `/ws/chats/...`.

```bash
cd openclaw-webchat
export OPENCLAW_MEDIA_LINK_SECRET="<long stable random secret>"
export OPENCLAW_MEDIA_LINK_TTL_SECONDS="3600"
export OPENCLAW_MEDIA_OUTBOUND_HOST_DIR="/path/on/host/to/openclaw/media/outbound"
export OPENCLAW_WEBCHAT_PUBLIC_BASE_URL="https://openclaw-webchat-api.example.com"
docker compose build
docker compose up -d
```

`OPENCLAW_WEBCHAT_PUBLIC_BASE_URL` is required when the frontend and backend use
different origins, so signed media links open on the bridge origin. Leave it
empty in all-in-one same-origin deployments.

See [Deployment](docs/DEPLOYMENT.md).

## Configuration

See [Configuration](docs/CONFIGURATION.md) and `config.example.json`.

Each authenticated user maps to:

- one OpenClaw instance;
- one OpenClaw agent ID;
- one canonical user key used in the OpenClaw `sessionKey`.

## Security

OpenClaw Gateway tokens and device identities are server-side only. The browser
only receives sanitized frames and signed media URLs.

See [Security Policy](SECURITY.md).
